# RoboSats Exp. Android build

The Android app bundles the experimental frontend and runs all coordinator,
Nostr, chat, and LNProxy traffic through an embedded Arti SOCKS5 proxy. The
WebView cannot make direct network requests.

## Requirements

- Java 17
- Android SDK 36 and build tools 36
- Android NDK `27.0.12077973`
- Rustup (the project pins Rust in `android/tor-native/rust-toolchain.toml`)
- `cargo-ndk`: `cargo install cargo-ndk --version 4.1.2 --locked`
- Node.js and npm

The native bridge currently locks `arti-client` and `tor-rtcompat` to the
`0.44.x` dependency graph in `android/tor-native/Cargo.lock`.

The native build ships `arm64-v8a` and `x86_64`. Gradle emits both ABI APKs and
a universal APK. Robot avatars are generated by the bundled `robohash` JNI
library for those same ABIs.

## Debug APK

From the repository root:

```bash
npm ci
npm run build:android:debug
```

Outputs are under `android/app/build/outputs/apk/debug/`.

## Release APK

```bash
npm run build:android:release
```

Outputs are under `android/app/build/outputs/apk/release/`.

## Build stages

1. `npm run build` compiles and optimizes the frontend.
2. `scripts/sync-android-assets.mjs` replaces the bundled WebView assets with
   the exact contents of `dist/`.
3. `android/tor-native/build-android.sh` compiles the locked Arti dependency
   graph into `libarti_android.so` for both supported ABIs.
4. `android/robo-identities-native/build-android.sh` builds the robot-avatar
   JNI library for both supported ABIs.
5. Gradle packages the web assets, JNI libraries, encrypted storage bridge,
   notification service, and Android shell.

The build also resolves Quartz's old ACINQ secp256k1 JNI dependency to
`0.23.0`; unlike the inherited `0.15.0` artifact, its arm64 binary supports
Android's 16 KB page size. Arti, secp256k1, and the RoboSats identity libraries
are packaged only for `arm64-v8a` and `x86_64`.

Generated frontend assets and native binaries are intentionally ignored. Every
APK build recreates them from committed source and lockfiles.

## Network boundary

- Arti bootstraps before the WebView is shown.
- The app binds SOCKS5 to localhost, normally `127.0.0.1:17392`.
- A single pooled OkHttp client handles HTTP and WebSocket traffic over SOCKS.
- `WebViewAssetLoader` serves bundled files from
  `https://appassets.androidplatform.net/`.
- Requests from the WebView to any other origin are rejected. User-initiated
  links are handed to an external app.
- Android cleartext transport is enabled because coordinator onion APIs use
  `http://`. Those requests are still accepted only by the native client and
  sent through Arti; the WebView cannot use the permission directly.
- Robot garage data and federation notification keys use Android encrypted
  shared preferences through a synchronous JavaScript bridge.

Never add a plain `fetch`, browser `WebSocket`, DNS lookup, or separate
`OkHttpClient` for Android-only network work. Use the transport adapters in
`src/domains/transport/androidBridge.ts` and `NativeNetworkClient`.
